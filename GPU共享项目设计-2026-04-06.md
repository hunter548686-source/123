# GPU 共享项目设计

日期：2026-04-06  
结论口径：基于官方文档、官方价格页与官方博客的最新公开信息；其中 Vast.ai 价格为其官方价格页调用的实时价格快照，Runpod 为官方价格页展示价，io.net 价格主要来自其官方对比文章，口径并不完全一致，比较时已单独标注。

## 1. 一句话结论

如果你的想法是“再做一个和 Vast.ai / Runpod / io.net 一样的通用 GPU 租赁平台”，**不建议直接做**。

如果你的想法是“把这三家的底层算力整合起来，做一个更稳定、更好用、更贴近具体场景的交付层”，**值得做，而且更现实**。

最推荐的方向不是“卖裸 GPU”，而是：

**做一个面向 AI 视频 / 图像 / 推理任务的多平台算力聚合与稳定交付平台。**

一句话理解：

`Vast.ai = 最便宜的床位`

`Runpod = 更稳定的酒店`

`io.net = 带代币经济的分布式酒店联盟`

`你要做的 = 帮用户自动订房、自动续住、自动换房、自动保成果的“旅行管家”`

## 2. 对标三家：最新比较

这里默认对标的“三家”是：**Vast.ai、Runpod、io.net**。  
原因：它们分别代表了三种最关键的模式：开放市场、托管云、DePIN 网络。

### 2.1 核心定位

| 维度 | Vast.ai | Runpod | io.net | 你的机会 |
| --- | --- | --- | --- | --- |
| 本质 | GPU 市场 | 托管 GPU 云 + 仍保留存量 Community Cloud | DePIN 分布式算力网络 | 多平台聚合与稳定交付层 |
| 供给来源 | 独立主机 / 机房卖家 | Secure Cloud 机房为主，Community Cloud 存量继续运行 | Worker / Supplier 网络，带 staking 与奖励 | 接入外部平台 + 自营保底节点 |
| 用户买的是什么 | 便宜 GPU | 更稳、更合规的 GPU | 成本更低的分布式算力 | “任务一定跑完”的结果交付 |
| 主要护城河 | 供给深度、低价 | 稳定性、合规、Serverless / Clusters | Token 激励、分布式调度 | 任务编排、断点续跑、失败切换、场景化产品 |

| 维度 | Vast.ai | Runpod | io.net |

| 计费 | 按秒 | 按秒 | 有 IO Credits / crypto / fiat，且有手续费口径 |

| 用户理解成本 | 低 | 低 | 高 |

### 2.5 对你最重要的战略信号

1. **Vast.ai 已经吃掉“开放长尾供给市场”**  
   如果你也做一个“大家来挂 GPU、别人来租”的网站，你会先撞上 Vast 的流动性优势。

2. **Runpod 正在往更托管、更合规的方向走**  
   它在 2026-02-24 正式说明：**不再接受新的 Community Cloud host**，说明它判断长期价值不在散户供给扩张，而在更稳的 Secure Cloud 与上层产品。

3. **io.net 的优势是网络叙事，不是简单产品体验**  
   它能吸引 crypto-native 供给，但也引入 staking、钱包、奖励、费用等复杂度。普通客户未必喜欢。

4. **因此“再做第四个平台”不是最佳切口**  
   更好的切口是：**利用这三家已经建好的底层供给，做一个更稳定、更简单、更懂场景的上层产品。**

## 3. 项目研判

## 3.1 不建议做的版本

### 项目形态

“全球 GPU 共享平台，用户上传机器，别人租用”

### 原因

1. **供给侧冷启动太难**  
   没有卖家，就没有便宜 GPU；没有买家，卖家也不会来。

2. **信任成本极高**  
   你要处理远程控制、计费、扣费、作弊、刷单、宕机、数据残留、争议仲裁。

3. **你会被迫先做‘脏活累活’**  
   可靠性评分、机器验证、自动退款、磁盘清理、作业中断恢复、KYC、税务、支付通道，这些才是平台本体。

4. **三家已经占住三种主流叙事**
   - Vast.ai：最便宜
   - Runpod：更稳更省心
   - io.net：DePIN / Token

### 结论

如果没有：

- 强供给资源
- 强流量入口
- 强区域政策优势
- 强垂直行业资源

就不要从“通用平台”开始。

## 3.2 建议做的版本

### 项目形态

**多平台 GPU 聚合 + 稳定交付层**

你不是卖“哪张卡”，你卖的是：

- 任务一定能跑完
- 成本自动最低
- 失败自动切换
- 用户不用懂 Vast / Runpod / io.net 差别

### 第一批适合的用户

1. AI 视频工作室
2. 做文生图 / 图生视频的独立开发者
3. 需要批量跑推理任务的小团队
4. 需要短期便宜 GPU、但不想自己盯机器的人

### 你真正卖的价值

不是 GPU，而是：

- **稳定性溢价**
- **自动化溢价**
- **结果交付溢价**
- **中文服务 / 本地化结算溢价**

## 4. 推荐项目定义

项目代号建议：**算力管家 / StableGPU / RenderRail**

### 4.1 产品定位

一个面向 AI 工作负载的 **GPU broker + orchestration layer**：

- 底层接 Vast.ai / Runpod / io.net
- 中间做库存聚合、价格比较、可靠性打分、任务调度
- 上层做任务产品化：视频、图像、LoRA 训练、批量推理

### 4.2 产品边界

#### 第一阶段不要做

- 开放 host 入驻
- 自己发 token
- 做全球裸机市场
- 做超复杂企业私有云

#### 第一阶段要做

- 多平台供给聚合
- 统一下单
- 任务队列
- 模型模板
- 断点续跑
- 失败自动迁移
- 成本统计
- 用户账单

## 5. MVP 方案

### 5.1 MVP 核心主张

**“同一个任务，自动帮你在 Vast / Runpod / io.net 里选最优机器，并保证失败后自动重试。”**

### 5.2 MVP 功能

1. **统一工作台**
   - 选择任务类型：视频生成 / 图像生成 / 模型推理 / LoRA 训练
   - 选择速度优先 / 成本优先 / 稳定优先

2. **资源路由器**
   - 拉取各平台库存与价格
   - 按规则排序：价格、稳定性、网络、历史成功率
   - 给出推荐实例

3. **任务编排器**
   - 启动实例
   - 拉起容器
   - 下载模型
   - 挂载存储
   - 执行任务
   - 上传产出
   - 释放实例

4. **中断恢复**
   - 周期性 checkpoint
   - 任务状态持久化
   - 机器失联后自动重排到下一台机器

5. **计费系统**
   - 记录底层真实成本
   - 加平台服务费
   - 展示每个任务的毛利

### 5.3 MVP 的正确用户故事

“我不关心底层在哪台 4090 上跑，我只关心 50 条视频今晚能不能出片。”

这才是你和 Vast / Runpod / io.net 的区分点。

## 6. 商业模式

### 6.1 收费方式

推荐按三层收费：

1. **底层算力成本 pass-through**
2. **平台服务费 15% - 35%**
3. **高级功能订阅**
   - 队列优先级
   - 自动重试次数
   - 团队协作
   - 结果归档
   - API 配额

### 6.2 为什么这个模式成立

用户通常不愿意为“GPU 小时”付高价，但愿意为下面几件事付钱：

- 不宕机
- 不丢任务
- 少操心
- 快交付

所以你要把价格标签从 `$ / hr` 改成：

- `$ / 任务`
- `$ / 结果`
- `$ / 月`

### 6.3 毛利逻辑

项目不靠“比 Vast 更便宜”赚钱，而靠：

- 自动化降低运维人工
- 多平台比价降低采购成本
- 风险池吸收偶发中断
- 场景封装提高客单价

## 7. 技术架构

### 7.1 系统分层

#### A. Provider Adapter 层

对接：

- Vast.ai API / CLI
- Runpod API
- io.net Cloud API

职责：

- 查询库存
- 创建实例
- 获取状态
- 销毁实例
- 读成本

#### B. Scheduling 层

职责：

- 价格评分
- 可靠性评分
- 区域选择
- 模型兼容性匹配
- 失败迁移决策

推荐评分公式：

`score = 价格权重 + 稳定性权重 + 启动时延权重 + 历史成功率权重`

#### C. Job Orchestrator 层

职责：

- 任务入队
- 分片执行
- checkpoint
- 重试
- 产物上传

建议技术：

- API：FastAPI / NestJS
- 队列：Redis + BullMQ / Celery
- 编排：Temporal / 自研轻量状态机
- 数据库：Postgres
- 对象存储：S3 兼容存储

#### D. Control Plane 层

职责：

- 用户、订单、余额
- 项目与任务管理
- 成本看板
- 账单
- 权限

#### E. Model Runtime 层

统一容器镜像：

- ComfyUI
- vLLM
- Diffusers
- LoRA 训练镜像

## 8. 核心算法与规则

### 8.1 资源选择策略

按任务类型选择资源：

- 视频生成：优先 4090 / 5090 / A100
- 批量图像：优先 4090 / 3090 / L40S
- 大模型推理：优先 A100 / H100 / L40S
- 训练：优先 A100 / H100

### 8.2 风险控制策略

1. **Vast.ai 默认不接长作业裸跑**
   - 超过阈值必须开启 checkpoint

2. **Runpod Secure Cloud 作为保底兜底池**
   - 当低价供应失效时自动切换

3. **io.net 适合弹性扩容，不适合一开始就做唯一生产底座**
   - 先作为价格补充池，而不是唯一主力池

4. **自营保底节点**
   - 至少保留少量自营 4090 / A100 节点用于 VIP 与紧急补单

## 9. 推荐切入顺序

### Phase 1：先卖“稳定”

目标：

- 不自己做 host 市场
- 只做用户侧统一入口
- 用外部平台供给完成交付

交付物：

- Web 控制台
- API
- 视频 / 图像模板
- 成本面板

### Phase 2：再卖“效率”

增加：

- 批任务
- 智能排程
- 队列优先级
- 团队空间
- 自动预算控制

### Phase 3：最后才碰“供给”

增加：

- 邀请优质 host 入驻
- 自建稳定池
- 私有保底集群
- 企业 SLA

这个顺序非常关键。

**先做需求侧聚合，后做供给侧平台。**

## 10. 竞争策略

### 10.1 不和三家正面打的点

不打：

- 最低价
- 最多 GPU 型号
- 最大 host 网络
- Token 叙事

### 10.2 你该打的点

打：

- 最省心
- 最适合具体任务
- 最稳定的跨平台交付
- 最清楚的成本控制
- 中文化 / 本地化体验

### 10.3 你的宣传语应该像这样

错误：

“全球 GPU 租赁平台”

正确：

“帮你自动调用全球最便宜且最稳的 GPU，把视频/推理任务稳定跑完。”

## 11. 最小团队配置

MVP 阶段 3 人足够：

1. 全栈 / 后端 1 人
2. 调度与 infra 1 人
3. 产品 / 增长 / 客服 1 人

如果只有 1-2 人，也能做，但一定要收缩范围：

- 先只做 1 个任务场景
- 先只接 2 个底层平台
- 先只支持 2-3 个模型模板

## 12. 风险清单

1. **供应波动**
   - 解决：多平台 + checkpoint + 保底池

2. **底层平台政策变动**
   - 解决：抽象适配层，不把业务绑死在单一平台

3. **任务失败导致赔付**
   - 解决：任务级 SLA，区分普通单与保底单

4. **毛利被价格战吃掉**
   - 解决：从卖算力切到卖结果和工作流

5. **用户绕过你直接去底层平台**
   - 解决：把价值放在调度、模板、稳定、批量管理，不是单次比价

## 13. 最终建议

### 建议做

**“多平台 GPU 聚合 + 稳定任务交付平台”**

### 不建议做

**“第四个通用 GPU 租赁 marketplace”**

### 为什么

因为三家已经把“供给组织方式”占完了：

- Vast.ai 占了开放市场
- Runpod 占了托管稳定云
- io.net 占了 DePIN 网络

你再做一个同类平台，很容易变成：

- 供给不够
- 买家不来
- 运维很重
- 价格还打不过

但如果你做“稳定交付层”，你能同时借力三家，而不是挑战三家。

## 14. 如果现在就开工，建议的 6 周版本

### 第 1-2 周

- 接 Vast.ai 与 Runpod
- 做统一实例查询接口
- 做统一任务模型

### 第 3-4 周

- 做任务执行器
- 做 checkpoint / 失败重试
- 做对象存储归档

### 第 5 周

- 做计费与毛利面板
- 做基础 Web 控制台

### 第 6 周

- 只上线 1 个强场景
  - 优先建议：视频生成批任务
  - 备选：图像生成工作室批量出图
