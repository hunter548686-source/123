日期：2026-04-06  
适用阶段：从 0 到 MVP，再到首个可收费版本  
目标：这份文档用于后续单开一个“执行对话”，让开发工作按计划一次推进

---

## 1. 项目定义

### 1.1 项目名称

暂定名：

- 算力管家
- StableGPU
- RenderRail

建议对外定位统一为：

**AI 视频任务稳定交付平台**

### 1.2 一句话定义

不是卖 GPU 租赁，而是帮用户自动调用底层算力平台，**稳定完成 AI 视频任务**。

### 1.3 核心价值

- 自动比价
- 自动选机
- 自动重试
- 失败迁移
- 结果归档
- 成本可视化

### 1.4 第一阶段边界

第一阶段只做：

- AI 视频任务
- 用户下单
- 调度底层 provider
- 执行任务
- 返回结果
- 账单闭环

第一阶段不做：

- GPU 供给侧 marketplace
- host 入驻
- token / DePIN
- 企业私有化
- 太多模型类型

## 2. 总体战略

### 2.1 战略原则

项目采用：

**上游借力，下游吃透，中间做厚**

即：

- 上游 GPU 资源先接第三方
- 中间调度和稳定交付自己做
- 下游客户、场景、交付和计费自己掌控

### 2.2 当前最合理的切入点

先做：

**多平台算力聚合 + AI 视频稳定交付**

而不是：

**第四个 GPU 租赁平台**

### 2.3 商业逻辑

用户不是为“某张 GPU 卡”付费，而是为下面这些东西付费：

- 任务跑完
- 节省人工盯单
- 更少中断
- 更清楚成本
- 更快交付

## 5. 技术总体结构

### 5.1 总体架构

建议采用四层结构：

1. 前端展示层
2. API 与业务层
3. 调度与任务执行层
4. 数据与存储层

### 5.2 推荐技术栈

#### 前端

- Next.js
- TypeScript
- Tailwind CSS

#### 后端

- FastAPI
- SQLAlchemy
- Pydantic

#### 数据

- Postgres
- Redis
- S3 兼容对象存储

#### 任务执行

- 异步 worker
- 状态机
- runtime executor

### 5.3 部署建议

鉴于你已经有 Vercel：

#### 第一阶段部署方案

- 前端：Vercel
- 轻量 API：可先随 Next.js 同仓
- 核心执行器和 worker：代码单独拆分，后续可迁出

#### 架构要求

无论第一阶段是否全放 Vercel，代码结构都必须支持：

- 前端独立
- API 独立
- worker 独立

也就是：

**部署可以先简单，代码结构不能写死。**

## 6. 代码仓库结构建议

```text
repo/
  apps/
    web/               # Next.js 前端
    api/               # FastAPI 后端
    worker/            # 异步任务执行器
  packages/
    shared/            # 类型、状态枚举、通用 schema
    ui/                # 公共 UI 组件
    config/            # 环境配置、常量
  infra/
    sql/               # 初始化 SQL
    docs/              # PRD、接口、架构文档
```

### 6.1 最低可行目录

如果先不做 monorepo，至少也要保留概念分层：

- `web`
- `api`
- `worker`
- `shared`

## 7. 页面与 UI 设计计划

## 7.1 UI 总原则

不要把产品做成“云服务器官网”，而要做成：

**任务交付型产品**

用户最关心的是：

- 任务是否完成
- 花了多少钱
- 失败怎么办
- 结果在哪里

### 7.2 视觉风格方向

建议风格：

- 偏专业 SaaS
- 不要极客感太重
- 重点体现稳定、效率、可信

建议关键词：

- 清晰
- 稳定
- 现代
- 有控制感

### 7.3 颜色建议

- 主色：深蓝 / 石墨黑 / 偏冷灰
- 强调色：电蓝或青绿
- 成功态：绿色
- 告警态：橙色
- 失败态：红色

避免：

- 过度花哨
- 游戏风
- 紫白默认模板感

### 7.4 字体建议

- 中文：思源黑体 / HarmonyOS Sans / 阿里巴巴普惠体
- 英文：Geist / IBM Plex Sans / Space Grotesk

### 7.5 重点页面设计要求

#### 首页

目标：转化

要求：

- 一眼看懂不是租 GPU，而是跑任务
- 明确平台价值
- CTA 清晰

#### 新建任务页

目标：下单

要求：

- 信息输入简洁
- 报价清楚
- 风险可解释

#### 任务详情页

目标：建立信任

要求：

- 状态清晰
- 进度可见
- provider 可见
- 重试记录可见

#### 后台任务页

目标：运营补单

要求：

- 快速看到异常任务
- 快速切换 provider
- 快速查看失败原因

### 7.6 UI 交付顺序

1. 信息架构
2. 低保真线框图
3. 高保真视觉稿
4. 组件清单
5. 前端实现

## 15. 风险与预案

## 15.1 技术风险

### 风险：provider 接口不稳定

预案：

- provider 适配层抽象
- 不把逻辑写死在单一平台

### 风险：任务执行时间过长

预案：

- worker 独立
- checkpoint
- 可迁移执行

### 风险：失败重试侵蚀利润

预案：

- 区分标准单与保底单
- 给每种策略设预算上限

## 15.2 产品风险

### 风险：做成“租 GPU 控制台”

预案：

- 始终以任务为中心
- 页面文案少谈显卡，多谈交付

### 风险：场景太多导致失焦

预案：

- 第一阶段只做 AI 视频

## 15.3 商业风险

### 风险：用户绕过平台

预案：

- 做稳定交付
- 做模板
- 做账单和归档
- 做运营支持

## 18. 最终结论

这个项目的正确推进方式不是：

- 先做一个很大的 GPU 平台

而是：

- 先做一个很强的任务交付产品

顺序一定是：

**场景 > 调度 > 稳定交付 > 商业闭环 > 供给延伸**

这份计划的目标，就是让你后面可以直接基于它开一个执行对话，把第一阶段完整做出来。
